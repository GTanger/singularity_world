# 決策：本專案採用插頭／插座語義（動詞－名詞交互）

**狀態**：已決斷  
**依據**：主管決斷；參考 `RUST(head_for_singularity)/doc/decisions/600_action_semantic_master_spec.md`，略去與本專案相左部分。

---

## 1. 決策內容

**本專案採用「插頭／插座」設計作為可執行性與交互的語義標準。**

- **插座 (Sockets)**：世界中的可互動對象（名詞）對外開放的「可執行動作」清單；每個名詞在資料／後端標註其具備哪些插座。
- **插頭 (Plugs)**：玩家或 NPC 的意圖，即**動詞**（如 Attack、Talk、Trade、Examine、Take、Drop）；對應 API 的動作類型。
- **匹配規則**：後端收到「動詞（插頭）+ 目標名詞」後，檢查目標是否具備對應**插座**；若匹配則執行邏輯並回傳結果或敘事，否則回傳不可執行。
- **與設計關鍵字對齊**：**全物件可互動**、**後端唯一數據真理**、**前端僅是顯示窗口**；可執行性一律由後端依插頭－插座判斷，前端不預測、不列舉可用動作（除依後端回傳之清單顯示）。

---

## 2. 名詞語義與邏輯插座 (Noun Semantics & Logic Sockets)

世界中的可互動對象由**語義標籤**與**邏輯插座**定義。**插座**即該對象對外開放的「可執行動作」清單；後端依此判斷請求是否合法。

### 2.1 活體實體 (Agent)

- **定義**：玩家、NPC 等具備狀態與行為的實體；本專案中**玩家＝NPC**，語義上皆為 `[Agent]`。**遊戲內任一角色都是實體**，彼此佔位、不可穿越。
- **實體與碰觸物理**：角色與角色碰觸時，產生**真實物理現象**——輕則推擠、重則撞開；後端須謹守此規則（位置、碰撞、推擠／撞開結果由後端計算）。屢次此類碰觸行為，可依設計觸發衝突或戰鬥（與決策 001 戰鬥規則對接）。
- **標籤**：`[Alive]`, `[Agent]`, `[Player]`, `[NPC]` 等。
- **常見插座**：`Talk`（對話）, `Look`（觀照）, `Attack`（攻擊）, `Trade`（交易）, `Invite`（邀請）等；依本專案實際需求增減。
- **後端職責**：管理狀態、位置、背包；碰觸時的推擠／撞開與後續衝突判定；響應動作指令並回傳結果或敘事。

### 2.2 建築／結構 (Structure)

- **定義**：城鎮或地圖中的建築、據點（如房屋、商店、工坊等），為**立於地圖上的實體物件**；**不設計進出建築**（無室內、無 Enter/Exit）。**有功能者**具備可執行動作清單（插座），玩家**接觸後點擊**即回傳可交互選項；無插座者為擋路實體。
- **標籤**：`[Structure]`, `[Interactable]`, `[Static]`；可細分 `[Shop]`, `[Tavern]`, `[Workshop]` 等。
- **常見插座**：`Examine`（調查／說明）, `Trade`（交易）, `Craft`（製作）, `Operate`（操作）等；**不含** `Enter`（進入）— 本專案不設計進出建築，見決策 003。
- **說明**：本專案世界為單一平面；結構可標註 sockets，後端依此回傳「接觸後點擊」時之可用功能清單；不可交互建築無插座，僅為擋路。

### 2.3 物品 (Item)

- **定義**：可拾取、攜帶、丟棄的物件。
- **標籤**：`[Item]`, `[Portable]`, `[Consumable]` 等。
- **常見插座**：`Take`, `Drop`, `Use`, `Examine`。
- **後端職責**：背包、物品增減、交易；由後端與資料為唯一真理來源。

### 2.4 其他可互動物件

- 地形上的採集點、機關、門等可依需求定義語義標籤與插座（如 `[Resource]` + `Mine`/`Chop`；`[Obstacle]` + `Examine`/`Operate`）；**不可僅憑文字創造物件**，須在後端／資料中有對應定義與插座。

## 3. 動詞邏輯（插頭）(Verb Logic — Plugs)

**動詞**即意圖，對應後端動作類型。後端收到「動詞 + 目標名詞」後，檢查目標是否具備對應**插座**；若匹配則執行邏輯並回傳結果或敘事，否則回傳不可執行。

### 3.1 即時性動詞（示例）

| 動詞 (Plug) | 常見目標／插座 | 後端邏輯（概要） |
| --- | --- | --- |
| **Attack** | `Attack` / `[Agent]` | 戰鬥系統（同一套規則，見決策 001）、寫 combat_log |
| **Talk** | `Talk` / `[Agent]` | 觸發對話／任務 |
| **Take** | `Take` / `[Item]` | 從場景／他人移入背包 |
| **Drop** | `Drop` / `[Item]` | 從背包移出至場景 |
| **Trade** | `Trade` / `[Agent]` 或 `[Structure]` | 金錢／物品交換 |
| **Examine** | `Examine` / 任意 | 回傳敘述／說明（可觀測距離點擊得敘述） |
| **Craft** | `Craft` / `[Structure]` 或物件 | 製作／祭煉等 |
| **Operate** | `Operate` | 操作機關／開關 |
| ~~Enter~~ | — | **本專案不採用**：不設計進出建築（決策 003） |

### 3.2 持續性意圖動詞（可選擴充）

- 如 **Mine** / **Chop**：需消耗精力、改變目標狀態；後端依 tick 或週期判斷中斷條件。本專案若未實作，可不列為預設插座。

## 4. 交互事件矩陣（插頭－插座匹配）

後端依「目標具備之插座」與「請求之動詞」判斷是否可執行，並執行對應邏輯。

| 動詞 (Plug) | 目標具備插座／語義 | 後端邏輯後果 | 回傳／敘事 |
| --- | --- | --- | --- |
| **Attack** | `Attack` / `[Agent]` | 戰鬥結算、寫 log | 戰鬥結果敘事（決策 001） |
| **Talk** | `Talk` / `[Agent]` | 觸發對話／任務 | NPC 對話或敘事 |
| **Take** | `Take` / `[Item]` | 物品入背包 | 「獲得了 [物品]」等 |
| **Drop** | `Drop` / `[Item]` | 物品離背包 | 「丟棄了 [物品]」等 |
| **Trade** | `Trade` / 商店或 Agent | 金錢／物品交換 | 交易結果 |
| **Examine** | `Examine` | 僅查詢 | 敘述文字 |
| **Craft** | `Craft` / 建築或物件 | 製作／祭煉 | 結果或錯誤訊息 |
| **Operate** | `Operate` | 操作機關 | 依定義 |
| ~~Enter~~ | — | 本專案不採用（無進出建築） | — |

**可用功能清單**：後端可依「當前目標物件之 sockets」回傳「該目標目前可執行的動詞清單」；前端僅在玩家選定目標時請求並顯示此清單，不自行列舉。

## 5. 後端與前端職責（與設計關鍵字一致）

### 5.1 後端

- **可執行性**：所有「能否執行」的判斷在後端；依據為「動詞（插頭）與目標名詞之插座是否匹配」及距離、狀態等規則。
- **可用功能列表**：依目標 ID 或類型，從資料讀取該目標開放的插座，回傳可顯示的動作列表。
- **指令處理**：收到「執行某動詞對某目標」時，先檢查目標是否具備對應插座，再執行既有邏輯（戰鬥、交易、對話等），回傳結果或敘事。
- **禁止**：不得允許「無插座」之目標執行該動詞；符合**不可無中生有**、**思維邏輯閉環**。

### 5.2 前端

- **可觀測 vs 可交互**：可依本專案定義（如視線內點擊得敘述；靠近且點擊才送交互指令）。
- **禁止**：前端不得自行列舉「可用的動詞」或猜測可執行性；一律依後端回傳的列表與結果。不得改動後端回傳的敘事語義。

---

## 6. 與決策 001（戰鬥規則統一）的關係

- **戰鬥**為插頭／插座設計下之一例：**Attack** 為插頭，目標需具備 **Attack** 插座（即 `[Agent]`）；匹配後走決策 001 之統一戰鬥規則。
- 其餘動詞（Talk, Trade, Examine, Take, Drop, Craft, Operate 等）同樣依「插頭－插座匹配」由後端判斷可執行性並執行；**Enter（進入建築）** 本專案不採用，見決策 003。
