# 討論紀錄：身份與職業分離 — 角色無身份綁定

> 日期：2026-02-12  
> 前提：延續 NPC 生成模板（`data/templates/README.md`）與人物生成機制的討論。

---

## 使用者觀點（記錄）

與目前模板設計相左的設想：

- **不該用職業綁定人，應該反過來。**
- 角色（不分玩家、NPC）**本身是無身份綁定的**。
- 舉例：持劍者在他人眼中是「劍客」，但他可能正在從事暗殺，也可能只是背著劍卻在務農。身份是**情境與他人認知**的結果，不是角色本質標籤。

### 既有四名角色的身份從何而來？

- 現有角色：兩位浮生客棧經理、兩位浮生客棧服務生。
- **誰給他們「經理／服務生」這個身份？**
  - 答：**客棧這個場所的主人**（目前尚無建立）。
- 場所主人 → 給予他們**工作（職業）**。
- 場所本身 → 給予他們在**外人眼中的身份**（經理、服務生）。
- **離開客棧後**：他們只是一般平民百姓，與他人無異。

### 小結（使用者設想）

| 層級 | 來源 | 作用 |
|------|------|------|
| 角色本體 | 無身份綁定 | 同一人可持劍、務農、受雇於客棧 |
| 工作／職業 | 場所主人 | 指派職務（經理、服務生等） |
| 外人認知的身份 | 場所＋在該場所的角色行為 | 「在客棧裡的那位是經理」 |

---

## 與現狀的對照

- **現狀（模板系統）**：archetype 以「職業」為單位（商人、鐵匠、經理…），NPC 生成時綁定一個職業 → 對應對話與行為模板。
- **使用者設想**：角色不綁職業；職業／身份由**場所＋場所主人**賦予，且僅在該情境下成立；同一實體在不同場所或無場所時可呈現不同或無特定身份。

---

## 身份／職業是否為插頭插座的一種？（2026-02-12 續）

**使用者提問**：不管是「場所主人給予工作（職業）」還是「場所給予身份（被觀測認同）」，是否就是插頭插座系統的一種？

**對照決策 002**：插頭＝動詞（意圖），插座＝名詞對外開放的「可執行動作」清單；匹配則可執行，否則不可。

**對齊方式**：

| 概念 | 插頭／插座解讀 |
|------|----------------|
| 場所主人給予工作 | **場所（或場所內的職務槽位）** 具備一組「職務插座」（如 Manage、Serve、Trade_at_counter）。主人**指派某人**擔任該職務 = 讓該人**插上**這個職務槽位 → 此人在該場所內**獲得**對應可執行動作（插座）。 |
| 場所給予身份（被觀測認同） | 同一組綁定：此人因插上該職務槽位，在該場所內**對外暴露**的插座即為「經理／服務生」的動作集；**外人觀測**到的就是「能做這些事的人」= 身份。身份即「在此情境下此人具備的插座集合」的觀測面。 |

**結論**：是同一套語義的延伸。  
- 既有插頭插座：**動詞（插頭）↔ 名詞的插座** → 能否對該對象執行該動作。  
- 身份／職業：**人（實體）↔ 場所＋職務（情境槽位）** → 在該情境下此人獲得哪些插座、被視為哪種身份。  
能力與身份皆來自**匹配／綁定**，不無中生有；「場所主人派職」與「場所給身份」可視為**情境化插座綁定**的一種。

## 對話模板 vs 動作模板：職業隨身、動作綁場所（2026-02-12 續）

**使用者觀點（記錄）**：

NPC 在獲得客棧場所主人的**經理任命（職業標籤）**，並被安排在**浮生客棧**工作（**身份／場所綁定**）之後：

- **對話模板**：仍具備**經理職業模板的對談權限**，可與別人在**任時任地**對談相關話題（例如在街上、在別處遇見熟人，仍可以經理身分聊客棧、訂房、價錢等）。
- **動作模板**：**相關動作模板**則必須**身在綁定場所**才可執行（例如辦理入住、定價、排班等，只能在浮生客棧內執行）。

| 模板類型 | 綁定依據 | 生效範圍 |
|----------|----------|----------|
| 對話模板 | 職業（任命） | 任時任地，與場所無關 |
| 動作模板 | 身份／場所綁定 | 僅在綁定場所內 |

---

## 後續可延伸

- 場所（Place/Venue）與場所主人（Owner）的建模。
- 「身份」作為「在場所內的職務＋外人認知」的衍生，而非角色固有屬性。
- 模板系統如何演進：由「職業綁人」改為「場所＋職務綁情境，人可擔任多情境角色」。
- 職務槽位與插頭插座的形式化：場所／角色各自暴露的插座、指派關係的資料結構。
- 模板拆分實作：對話模板依職業（任命）全域生效；動作模板依場所綁定，僅在該場所內可執行。

## NPC 生成模板應如何拆分？公共模板是否需獨立？（2026-02-12 續）

**使用者觀點（記錄）**：

- NPC 生成應與玩家一致：**每位 NPC 生成時只具備萬人萬相的 seed**，職業與身份**並非生成當下即綁定**。
- 因此以下應拆分為**獨立的數個檔案**：
  - **職業動作模板**
  - **職業對話模板**
  - **NPC 生成模板**（僅負責「從 seed 生出一個無身份綁定的實體」，不綁職業／身份）

**使用者提問**：

- 是否應有**非職業的公共對話模板、公共動作模板**？
- 還是這其實由**插頭插座系統**就能設定？

**整理與對齊**：

| 項目 | 說明 |
|------|------|
| NPC 生成模板 | 只做「實體生成」：輸入 seed（萬人萬相），輸出一個無職業／無場所身份綁定的角色；與玩家創角同邏輯。 |
| 職業對話模板 | 依職業類型分檔（如經理、服務生、商人）；當角色**被任命該職業**時掛載，任時任地可用。 |
| 職業動作模板 | 依職業類型分檔；當角色**在綁定場所內**時，才開放對應插座與執行邏輯。 |
| 公共對話模板 | 任何角色都有的通用對談內容（招呼、閒聊、拒絕、無特定職業時的回應）；可獨立一檔或一組檔，避免「沒職業就不會說話」。 |
| 公共動作模板 | 即決策 002 的 **Agent 預設插座**（Talk, Look, Attack, Trade 等）：任何活體實體與生俱來的能力。若插座清單由後端／schema 寫死，可不必另建「公共動作模板檔」；若希望資料驅動，可有一份 **default_sockets** 或 **public_actions** 清單。 |

**結論（供決策）**：

- **生成**：獨立「NPC 生成」邏輯／檔，只產帶 seed 的實體；職業／身份由後續指派綁定。
- **職業模板**：職業對話、職業動作分檔，依職業類型與場所綁定掛載。
- **公共對話**：建議有獨立「公共對話模板」（或一組預設句庫），讓無職業或情境外的角色仍有可用的 Talk 內容。
- **公共動作**：本質即插頭插座系統裡「Agent 預設具備的插座」；可維持在 schema／程式定義，若希望全資料驅動再抽成一份公共動作清單檔。

## 實作面提議與回饋（2026-02-12）

以下為對目前設想的補充建議，供實作時取捨。

### 1. 場所（Venue）的範圍要怎麼算「在場」

動作模板只在**綁定場所內**生效，需要明確定義「在場」：

- **作法 A**：場所對應 `rooms.json` 的 **zone**（例如「浮生客棧」= 該 zone 下所有房間）。判斷時用 `current_room.zone === venue.zone_id`。
- **作法 B**：場所帶一組 **room_ids**，僅列出的房間算在場。
- 建議：若目前 zone 已能表達「一間客棧／一棟建築」，優先沿用 zone，少一層對應表；若未來有「同一 zone 多個子場所」再改為 room_ids 或 venue→rooms 對照表。

### 2. 一人多職業時，對話怎麼併

同一人可被任命多個職業（例如甲客棧經理＋乙商行顧問），對話模板會來自多個職業。若兩邊都定義了同一 key（例如 `greet`），需要規則：

- 選一個：例如「最近任命的職業優先」或「依職業 priority 欄位」。
- 或合併：例如 greet 隨機抽任一個職業的 greet，或依情境選（對方在甲客棧就偏經理口吻）。
- 第一版建議：**合併池**，同一 key 把所有職業的句子都丟進池裡隨機抽，簡單且不丟內容；若日後要區分「主職／副職」再加 priority。

### 3. 場所主人尚未建立時的過渡

現有四位 NPC 已有經理／服務生身份，但「客棧主人」尚未建模。建議：

- 指派關係可先允許 **assigned_by 為空或 system**，只記錄「誰、什麼職業、哪個場所」；有主人實體後再補上 assigned_by。
- 這樣既有四名 NPC 可先遷移為：實體（含 seed）＋指派（經理／服務生，場所=浮生客棧），不卡在「等主人上線」。

### 4. 檔案與資料結構建議

- **職業定義**：可保留類似現有 archetypes 的**職業型別表**（如 `occupations.json`），但只描述「這種職業用哪個對話檔、哪個動作檔」，不帶 spawn_weight；生成時不從這裡選「誰當經理」，而是由指派寫入「誰被派了經理」。
- **NPC 生成**：「從 seed 生出實體」多半是程式邏輯＋少量設定（預設插座、從 seed 推的欄位）；不必一個巨大的「生成模板檔」，可一份小 config（例如 default_sockets、seed→欄位對照）即可。
- **公共對話**：單一 `public_dialogue.json` 即可，與各職業對話同結構（greet、idle、talk…），Talk 時先查職業模板再 fallback 公共。

### 5. 既有資料的遷移

- 現有 2 經理、2 服務生：為每人補 **seed**（若無則用 id 或名字 hash 成 deterministic seed），建立 **Venue「浮生客棧」**（對應現有 zone 或 room 列表），寫入 **Assignment**（entity_id, occupation=經理/服務生, venue_id）。  
- 對話／行為仍用現有經理／服務生模板，只是改為「因指派而掛載」而非「生成時綁死」。

### 6. 與插頭插座的對接點

- **列可執行動作**：後端依「此人當前房間是否在任一指派場所內」決定是否加上該職業的動作插座；預設 Agent 插座永遠在。
- **執行動作時**：同樣檢查「此動作是否為職業動作且當前在綁定場所」；若不在場則回傳不可執行或 fallback 敘事（例如「你現在不在櫃台」）。

以上不改變你設定的「生成只帶 seed」「職業／身份事後綁定」「對話隨職業、動作綁場所」；僅在實作層面補齊邊界與過渡，方便落地。
