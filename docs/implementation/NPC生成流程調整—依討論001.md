# NPC 生成流程調整 — 依討論 001 共識

> 對齊：[討論紀錄 001 身份與職業分離—角色無身份綁定](../discussions/001_身份與職業分離—角色無身份綁定.md)

---

## 一、現狀簡述

| 環節 | 目前作法 |
|------|----------|
| **生成** | `db/npc.go`：`InsertNPC(id, displayChar, gender, displayTitle)` 建立實體時即寫入 `display_title`（如「經理」「服務生」）；`SeedNPCs` 用硬編碼 `defaultNPCs` 一次寫死 id、title、work_room、rest_room、排班。 |
| **身份來源** | 職稱與排班綁在「建立當下」，無場所、無指派關係、無主人。 |
| **行為／對話** | `db/behavior.go` 讀取 `data/npc_behaviors.json`，以 **職稱（title）** 為 key（經理、服務生）取 Idle、EnterReactions、Shift*、Wander*；`GetNPCTitle` 直接查 `entities.display_title`。 |
| **模板** | `data/templates/`（archetypes、dialogues、behaviors）**尚未被任何 Go 程式讀取**；程式只依賴單一 `npc_behaviors.json`。 |

---

## 二、調整後目標（依共識）

- **生成**：只產「帶 seed 的實體」，不寫職業／身份；與玩家創角同邏輯（seed → 屬性／穿搭）。
- **職業與身份**：由**指派（Assignment）** 決定——誰、什麼職業、哪個場所；`display_title` 改為由「當前指派」推導或快取，而非寫死欄位。
- **對話模板**：依「此人被指派的職業」掛載，任時任地可用；來源改為 `data/templates/dialogues/`（公共 + 職業）+ 關鍵字檢索（後續實作）。
- **動作模板**：依「此人被指派的職業 + 當前房間是否在該場所的 room_ids 內」掛載，僅在綁定場所內生效；來源可為 `data/templates/behaviors/` 或沿用 npc_behaviors 結構但改為依職業＋場所解析。
- **場所**：新增 **Venue**（id, name, room_ids）；「在場」＝當前房間在 venue.room_ids 內。

---

## 三、具體調整步驟

### 3.1 資料與設定

1. **新增資料表或設定檔**
   - **venues**：`id, name, room_ids`（room_ids 為 JSON 陣列或另表 venue_rooms(venue_id, room_id)）。
   - **assignments**：`entity_id, occupation_id, venue_id, assigned_by`（assigned_by 可空，過渡用）。
   - **occupations**（表或 `data/templates/occupations.json`）：`occupation_id, name, dialogue_file, action_file`，不帶 spawn_weight。

2. **職業型別與模板檔對應**
   - 新增 `data/templates/occupations.json`（或同義），列出職業 id、名稱、對應的 dialogue 檔、action/behavior 檔。
   - 現有 `data/templates/dialogues/`、`data/templates/behaviors/` 改為依職業 id 掛載；可保留 `data/npc_behaviors.json` 作為「經理／服務生」的動作＋敘事來源，改由 occupation 指向（例如 occupation 經理 → npc_behaviors 的 roles.經理），或逐步把內容遷到 `behaviors/manager.json` 等。

3. **公共對話**
   - 新增 `data/templates/dialogues/public_dialogue.json`（或同路徑），Talk 時無職業或無命中時 fallback。

### 3.2 生成流程（db/npc.go）

1. **InsertNPC**
   - 簽名改為只接受與「實體本身」相關參數：`id, displayChar, gender`（或由 seed 推）；**不再接受 displayTitle**。
   - 寫入 `entities` 時不寫 `display_title`，或寫空字串／預設「行人」；其餘同現有（soul_seed, vit/qi/dex from seed, equipment_slots）。

2. **SeedNPCs（遷移＋預設四名）**
   - 若 DB 已有舊資料：為既有四名 NPC 補上 seed（若無則用 id 的 hash 成 deterministic seed），不刪除現有 `display_title`（可保留一段時間供過渡）。
   - 建立 **Venue「浮生客棧」**：id 如 `venue_life_inn`，room_ids 為浮生客棧相關房間 id 列表（life_hall, life_dining, ...）。
   - 建立 **Assignment**：陳正明→經理＋浮生客棧，林小雯→服務生＋浮生客棧，張明德→經理＋浮生客棧，王阿財→服務生＋浮生客棧；assigned_by 可空。
   - 之後新 NPC：只呼叫 `InsertNPC`（無 title），不寫 npc_schedules；排班可改為「有 assignment 時，依 venue.room_ids 與 shift 決定 work_room/rest_room」，或暫時保留 npc_schedules 僅給既有四名用。

### 3.3 身份與標題解析

1. **GetNPCTitle / 對外顯示用標題**
   - 改為：查詢該 entity 的 **assignments**（可含 venue 名稱），若有則回傳「該指派對應的 occupation 名稱」（如經理、服務生）；若多職可取第一個或約定規則。
   - 過渡期：若 assignments 無資料，fallback 查 `entities.display_title`（相容舊資料）。

2. **行為／對話解析**
   - 依 **entity_id** 查 assignments → 得到 occupation_id(s) → 依 occupations 表／檔載入對應 dialogue 與 action 模板。
   - **對話**：合併公共 + 所有指派職業的對話池；任時任地可用。
   - **動作**：僅當 `current_room` 屬於該 assignment 的 venue.room_ids 時，才掛載該職業的動作模板（插座＋執行邏輯）；否則僅預設 Agent 插座。

### 3.4 行為載入（db/behavior.go 或新模組）

1. **載入來源**
   - 啟動時載入 `data/templates/occupations.json`、`data/templates/dialogues/public_dialogue.json`、各職業 dialogue/behavior 檔（或沿用 npc_behaviors.json 按職業 key 取）。
   - `PickIdleEmote`、`PickEnterReaction` 等改為接受 **entity_id**（或 entity_id + current_room）：內部查 assignments → occupation(s) → 再依 occupation 取模板；若需「在場」才有的敘事，可依 current_room 是否在 venue 內決定是否採用該職業的該類內容。

2. **排班與移動**
   - **npc_schedules**：可保留給「定點排班 NPC」使用；若未來排班改為「由 assignment + venue + shift 推導」，可再改為由 venue.room_ids 與設定決定 work_room/rest_room。第一版可維持現有 ApplySchedules 邏輯，僅把 GetNPCTitle 改為自 assignments 推導。

### 3.5 插頭插座對接

1. **列可執行動作**
   - 後端依「當前房間是否在任一指派場所的 room_ids 內」決定是否為該 NPC 加上該職業的動作插座；預設 Agent 插座（Talk, Look, Attack, Trade 等）永遠在。

2. **執行動作時**
   - 若動作為職業動作（如辦理入住），檢查「此 NPC 是否在綁定場所內」；若不在則回傳不可執行或 fallback 敘事（如「你現在不在櫃台」）。

## 四、檔案與程式對照

| 項目 | 調整 |
|------|------|
| `db/npc.go` | InsertNPC 移除 displayTitle；SeedNPCs 改為建 Venue、Assignment，四名 NPC 補 seed＋指派。 |
| `db/schedule.go` | GetNPCTitle 改為自 assignments 推導（可新函式 GetNPCTitleFromAssignments）；過渡期 fallback display_title。 |
| `db/behavior.go` | 改為依 entity_id → assignments → occupation → 模板；或新增 db/assignment.go、db/venue.go 查詢，behavior 改為接受 entity_id + room_id。 |
| 新增 | venues、assignments 已由 store（data/venues.json、data/assignments.json）提供；`db/occupation.go` 或讀取 occupations.json；載入 templates 的 dialogue/action 依職業。 |
| `data/templates/` | 新增 occupations.json、public_dialogue.json；dialogues/、behaviors/ 依職業 id 對應。 |
| `main.go` | 啟動時除 LoadBehaviors 外，可增加載入 venues、occupations、templates（或延後到第一次需要時）。 |

---

## 五、實作順序建議

1. **Phase 1**：資料層 — 建 venues、assignments 表與 migrations；新增 occupations.json；SeedNPCs 改為寫入 Venue「浮生客棧」與四筆 Assignment；InsertNPC 不再寫 display_title（或寫空），改為可選「遷移時保留舊 display_title」。
2. **Phase 2**：解析層 — GetNPCTitle 改為自 assignments 推導，fallback display_title；PickIdleEmote 等改為依 entity_id（＋room_id）查 assignment → occupation → 既有 npc_behaviors 或 templates。
3. **Phase 3**：模板拆分 — 對話／動作改為從 data/templates/dialogues/、behaviors/ 依職業載入；公共對話 fallback；關鍵字檢索可列為後續。
4. **Phase 4**：插座與執行 — 列可執行動作、執行動作時檢查「在場」與職業動作。

此順序可讓「身份與職業分離」先落地，再逐步把模板與關鍵字對話補齊。

- **data/templates/occupations.json**：職業型別表，含 經理／服務生，欄位 name、dialogue_file、behavior_file、action_sockets（目前為空陣列，之後可加 Trade 等）。

- **列可執行動作**：sendRoomView 改為接受 database，對 NPC 使用 `db.GetSocketsForNPC(database, e.ID, roomID)` 作為 ve.Actions；玩家仍用 e.Sockets()。
- **do_action 執行**：目標為 NPC 時以 GetSocketsForNPC 取得有效插座並 HasSocket 檢查；若動作非預設插座（IsDefaultSocket），再檢查 `EntityInVenueAtRoom(database, targetID, playerRoom)`，不在場則回傳「對方此刻不在工作場所，無法執行……」。
