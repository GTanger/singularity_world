# 技術約束規則（Singularity World）

本文件為**程式開發階段**的技術與協作約束，所有組員（含 AI）撰寫或修改程式碼時須遵守。與 [COLLABORATION](COLLABORATION.md) 協作約定並行。

**AI 進行專案索引或檢索時，須先閱讀 [協作約定](COLLABORATION.md) 與本文件，再依 [文檔索引](文檔索引.md) 查找設計與規格。**

---

## 語言與技術棧

- **後端**：Go（標準庫優先，第三方庫須經主管同意才可引入）
- **前端**：HTML + CSS + JavaScript（原生，不用框架，除非主管同意）
- **資料庫**：SQLite（使用 go-sqlite3 或 modernc.org/sqlite）
- **通訊**：WebSocket（gorilla/websocket 或標準庫）

---

## 程式碼風格

- 每個函數開頭必須有一段**中文註解**，說明：這個函數做什麼、吃什麼參數、回傳什麼、會產生什麼副作用
- 變數命名用英文，但**關鍵業務邏輯旁須附中文行內註解**
- 一個函數不超過 **50 行**，超過就拆
- 禁止巢狀超過 **3 層**的 if/for，超過就重構
- 不用 interface 除非有**兩個以上的實作**需要它
- **禁止 init() 函數**
- 所有錯誤必須**明確處理**，禁止 `_ = someFunc()`

---

## 檔案結構

- 每個 `.go` 檔案開頭須有**檔案級中文註解**，說明這個檔案負責什麼
- 一個檔案只做一件事，不超過 **300 行**
- 修改任何檔案前，先說明：**要改哪個檔案的哪個函數、為什麼改、改完預期行為是什麼**

---

## 協作流程

- 寫新模組前，先用**中文列出設計方案**，經主管同意後才寫程式碼
- 每次提交的程式碼必須**能編譯通過**
- **不得自行引入新依賴**，須先提出理由供主管決斷
- **不得自行刪除或重構既有程式碼**，須先說明理由
- 遇到設計抉擇時，提供**至少兩個方案及利弊**，由主管選擇

---

## 禁止事項

- 禁止產生**框架式空殼程式碼**（不可以生成一堆 TODO 或 placeholder）
- 禁止**未經討論就改變資料結構**
- 禁止在程式碼中**硬編碼遊戲數值**，所有可調參數必須集中管理
- 禁止**省略錯誤處理**
