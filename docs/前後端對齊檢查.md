# 前後端代碼對齊檢查

## 後端

### 1. 地圖載入 `world/chunk_load.go`
- **格式**：151 行 × 每行 151 個非空白 rune（跳過 `unicode.IsSpace`），對應區塊內 `(x, y)` = (第 C 字, 第 L 行)。
- **儲存**：`g.Cells[y*151+x] = TerrainFromRune(r)`，row-major。
- **未對應 rune**：一律當草（`TerrainFromRune`）。
- **結論**：與 `world/map.go`、`data/maps/README.txt` 一致。

### 2. 格點與地形 `world/map.go`
- **Grid**：`Cells[y*Width+x]`，Width=Height=151。
- **At(x,y)**：越界回傳牆。
- **結論**：與 chunk_load 的 `y*151+x` 一致。

### 3. 區塊視野 `game/chunk_view.go`
- **ChunkRows()**：扁平 `[]string`，長度 151×151，索引 = `ly*151+lx`（先 row 再 column）。
- **順序**：y=0 的 151 格 → y=1 的 151 格 → … → y=150 的 151 格。
- **結論**：與 `world.Grid` row-major 一致；協議註解正確。

### 4. 區塊常數 `game/zone.go`
- **ChunkSize** = 151；ChunkIndex / ChunkBounds / InChunk 皆用 151。
- **結論**：與 chunk_load、chunk_view、前端 151 一致。

### 5. 協議 `server/protocol.go`
- **ViewMsg.Rows**：`[]string`，`json:"rows"`，註解「扁平 151×151，索引 ly*151+lx」。
- **結論**：與 ChunkRows 回傳型別一致。

### 6. 處理流程 `server/handler.go`
- **sendView**：`Rows: view.ChunkRows()`，直接傳扁平陣列。
- **結論**：無額外轉換，與協議一致。

## 前端

### 1. 視野與繪製 `web/canvas.js`
- **VIEWPORT_W/H**：`floor(canvas.width/30)`，4530→151。
- **格式判斷**：
  - **扁平**：`rows.length >= 22801` 且 `typeof rows[0] === 'string'`（後端送 22801 個單字串，符合）。
  - **巢狀**：`rows[0]` 為陣列且長度 ≥ 151（相容舊格式）。
  - **字串行**：`rows[0]` 為長度 ≥ 151 的字串（相容舊格式）。
- **取字**：扁平時 `ch = rows[ly*151+lx]`，與後端索引一致。
- **繪製**：`cellPx = dx*CELL_SIZE`，`cellPy = dy*CELL_SIZE`，一格 30px，無 *3 或錯位。
- **結論**：與後端扁平 151×151、索引 ly*151+lx 對齊。

### 2. 狀態與 WS `web/main.js`
- **view**：`state.rows = Array.isArray(msg.rows) ? msg.rows : []`，再傳給 `canvasDrawChunkView(state.cx, state.cy, state.rows, ...)`。
- **結論**：收到之 `rows` 與協議一致，直接給 canvas 使用。

## 建議

1. **地圖檔**：`data/maps/0_0.txt` 每行應為 **151 個地形字**（可含空格，載入時會跳過），不要多出隱藏/未知字元，否則多餘字元會變成草、易出現「川草草」或整片草。
2. **重啟**：改過 `chunk_load.go` 或任一後端後需重新 `go build` 並重啟程式，前端強重整（Ctrl+Shift+R）以載入最新 JS。
