# 房間非人物件互動系統

> 最後更新：2026-03-03  
> 狀態：已實作（11.1～11.7）；歸屬權／偷竊／有狀態物件為後續階段  
> 依賴：插頭插座語義（決策 002）、NPC 活化系統、房間制

---

## 一、概述

目前房間制下，玩家的互動對象僅限於「人」（NPC / 玩家），房間描述雖然提到了宮燈、匾額、蒸籠、陶甕、捲軸等物件，但這些全是純文字裝飾，不可觸碰、不可操作。

**本系統的目標**：讓房間中的非人物件成為可互動的一等公民，與角色共享同一套插頭插座語義，並引入歸屬權機制，使互動產生真實的社會後果。

**核心原則**：

- **描述即物件**：房間描述裡出現的可互動物件，在描述文字中以富文字標記（色彩 + 可點擊），玩家一眼即知「這東西可以摸」。
- **插頭插座統一**：非人物件與角色使用相同的 `do_action` 流程，後端判斷可執行性，前端不猜測。
- **歸屬權真實**：有主之物受保護，無主之物自由取用，偷竊需四下無人，被發現有後果。

## 二、前端 UI 設計

### 2.1 房間描述富文字標記

現有的 `formatDesc()` 已將 `【文字】` 渲染為綠色 `.desc-highlight`。非人物件將延伸此機制：

**規則**：房間描述中，可互動物件以 `〔物件名〕` 標記（注意使用全形方頭括號，與 NPC 名字的 `【】` 區分）。

**渲染效果**：

| 標記 | 用途 | 前端樣式 |
|------|------|---------|
| `【名字】` | 人物名（已有） | 綠色 `.desc-highlight` |
| `〔物件名〕` | 可互動物件 | 琥珀色 `.desc-object`，帶底線虛線，hover 變亮 |

**範例**：

```
原始描述（rooms.json）：
"挑高的大廳中央懸著一盞〔六角宮燈〕，暖黃的光暈映在兩側的木柱上。
櫃台後方掛著一面寫有【浮生】二字的〔匾額〕，墨跡蒼勁。
角落立著一塊〔佈告欄〕，上頭釘滿了紙條。"

渲染結果：
「六角宮燈」「匾額」「佈告欄」以琥珀色顯示，玩家可直接點擊。
```

**點擊行為**：玩家點擊描述中的物件名 → **直接送出「觀看」**，觀看敘事出現在右側 Log；下一行顯示該物件的其他可執行動作（如【閱讀】【嗅聞】），點擊即送對應 `do_action`。

### 2.2 點擊即觀看 + Log 下一行其他動作（現行實作）

物件與人物採用同一套互動邏輯，不另設浮動下拉或手風琴。

**互動流程**：

```
1. 玩家點擊描述中的 〔匾額〕 或人物欄的「經理」
2. 前端立即送出 do_action { entity_id: ..., action: "Look" }
3. 伺服器回傳 action_result，觀看敘事 append 到 Log
4. main.js 依 target_id 從 state.objects 或 state.entities 取得其他動作（排除 Look），
   在敘事下一行 append 可點擊的【閱讀】【對話】【攻擊】等
5. 玩家點擊 Log 中的【閱讀】→ 送出 do_action Read → 結果再 append 到 Log
```

**人物欄**：同房 NPC/玩家以列表顯示（▸ 名稱），點擊即送觀看，不再有手風琴展開動作列；其他動作（對話、攻擊）同樣在 Log 觀看敘事的下一行顯示。

**樣式對照**：

| 元素 | 說明 | 色彩 |
|------|------|------|
| `〔物件名〕` 常態 | 琥珀色 + 虛底線、可點擊 | `#ffb74d` |
| `〔物件名〕` hover | 亮琥珀色 + 實底線 | `#ffa726` |
| Log 內其他動作 | `.log-object-action`，可點擊 | `#ffb74d`，hover `#ffa726` |

### 3.2 欄位定義

| 欄位 | 型別 | 必填 | 說明 |
|------|------|------|------|
| `id` | string | 是 | 全域唯一識別符，格式 `{room_id}_{物件代號}` |
| `name` | string | 是 | 顯示名稱，同時也是房間描述中 `〔〕` 內的文字 |
| `owner` | string | 是 | 歸屬權標記（詳見 §四），空字串 = 無主 |
| `owner_guard` | string[] | 否 | 看守者職稱列表，這些 NPC 在場時會阻止未授權使用 |
| `sockets` | string[] | 是 | 可用動詞（插座），後端判斷可執行性 |
| `responses` | map | 是 | 動詞 → 敘事文字（基礎回應） |
| `use_blocked` | string | 否 | 有主物件被看守者阻止時的敘事，`{guard}` 為佔位符 |
| `use_steal` | string | 否 | 四下無人時偷用的敘事前綴 |
| `ambient` | string[] | 否 | 環境敘事池，物件可產生 ambient narrate |
| `state` | string | 否 | 有狀態物件的初始狀態（如 `"locked"` / `"open"` / `"cold"` / `"hot"`） |
| `state_responses` | map | 否 | 狀態 → 動詞 → 敘事覆蓋（狀態不同，互動結果不同） |

### 3.3 動詞（插座）列表

沿用決策 002 的動詞體系，針對非人物件常用的：

| 動詞 | 說明 | 典型目標 |
|------|------|---------|
| `Look` | 仔細端詳外觀 | 所有物件 |
| `Read` | 閱讀文字內容 | 匾額、佈告欄、書籍、捲軸 |
| `Use` | 操作/使用 | 灶台、井、磨刀石、織布機 |
| `Open` | 開啟 | 箱子、門、抽屜 |
| `Sit` | 坐下 | 椅子、長凳、蒲團 |
| `Smell` | 嗅聞 | 香爐、酒甕、花草 |
| `Taste` | 品嚐 | 酒甕、食物 |
| `Take` | 拾取/摘取 | 野花、草藥、掉落物 |
| `Chop` | 砍伐 | 樹木、柴堆 |
| `Operate` | 操作機關 | 機關、暗門、絞盤 |

## 四、歸屬權模型

### 4.1 歸屬權分類

```
歸屬權
├── 無主（owner: ""）
│   ├── 公共設施：佈告欄、路標、官井 → 任何人自由互動
│   └── 野生資源：花草、樹木、礦石 → 自由採集/砍伐
│
├── 有主（owner: "inn" / "npc_id" / "faction_id"）
│   ├── 可觀看：Look / Read 不受限，所有人可觀看有主之物
│   ├── 需授權才可使用：Use / Open / Take 等操作性動詞需檢查權限
│   │   ├── 看守者在場 → 被制止（use_blocked 敘事）
│   │   ├── 看守者不在場、其他人在場 → 行為被目擊（有後果）
│   │   └── 四下無人 → 可偷用（use_steal 敘事，觸發偷竊事件）
│   └── 可租借：owner_rentable = true → 付費後獲臨時使用權
│
└── 玩家所有（owner: "player:{player_id}"）
    └── 未來擴充：玩家擁有的房產內之物件，他人互動同上規則
```

### 4.2 權限判定流程

```
玩家對有主物件執行操作性動詞（Use / Open / Take / Chop …）
│
├─ 動詞是 Look / Read / Smell？
│  └─ 直接通過，回傳 responses 敘事（觀看不受限）
│
├─ owner 為空？
│  └─ 直接通過，回傳 responses 敘事（無主之物）
│
├─ 玩家擁有授權（租借/擁有者本人）？
│  └─ 直接通過
│
├─ 檢查同房是否有 owner_guard 中的 NPC
│  ├─ 有看守者在場 → 阻止，回傳 use_blocked 敘事
│  │   （看守者 NPC 產生對應的 narrate 反應）
│  │
│  ├─ 無看守者，但有其他 NPC 或玩家在場
│  │  └─ 行為被目擊 → 回傳偷竊敘事 + 記錄事件
│  │     （目擊者 NPC 可能產生反應：報警、喊叫、冷眼旁觀）
│  │
│  └─ 四下無人（房間內僅自己）
│     └─ 偷用成功 → 回傳 use_steal 敘事 + 記錄偷竊事件
│        （事件進入日誌，可能影響後續 NPC 對你的態度）
```

### 4.3 「四下無人」判定

「四下無人」= 同房間內除操作者外，沒有任何其他實體（NPC 或玩家）。

```go
func isAlone(database *sql.DB, roomID, playerID string) bool {
    entities := db.GetEntitiesInRoom(database, roomID)
    for _, e := range entities {
        if e.ID != playerID { return false }
    }
    return true
}
```

### 4.4 偷竊後果（設計方向）

| 情境 | 後果 |
|------|------|
| 四下無人，偷用成功 | 事件記入日誌；物主方定期巡查，累積偷竊事件可能觸發追緝 |
| 有其他人目擊 | 事件記入日誌 + 目擊者記憶；NPC 可能當場反應（喊叫/報警） |
| 看守者在場被阻止 | 無實質後果，僅敘事；但頻繁嘗試可能降低好感度 |

```go
    Responses      map[string]string `json:"responses"`
    UseBlocked     string            `json:"use_blocked"`
    UseSteal       string            `json:"use_steal"`
    Ambient        []string          `json:"ambient"`
    State          string            `json:"state"`
    StateResponses map[string]map[string]string `json:"state_responses"`
}

var roomObjects map[string][]RoomObject  // room_id → []RoomObject
```

### 5.2 協議擴充

`RoomViewMsg` 新增 `Objects` 欄位：

```go
type ViewObject struct {
    ID      string   `json:"id"`
    Name    string   `json:"name"`
    Actions []string `json:"actions,omitempty"`
}

type RoomViewMsg struct {
    // ...現有欄位...
    Objects []ViewObject `json:"objects,omitempty"`
}
```

### 5.3 `handleDoAction` 擴展

```
收到 do_action { entity_id: "life_hall_board", action: "Read" }
│
├─ 先查角色（現有流程）→ 找不到
├─ 再查物件（新增）→ 找到 life_hall_board
├─ 檢查物件是否在玩家同房間
├─ 檢查 HasSocket
├─ 檢查歸屬權（§4.2 流程）
└─ 回傳 action_result 敘事
```

### 5.4 物件環境敘事

物件的 `ambient` 欄位可被主迴圈的 idle tick 使用，與 NPC 閒置動作混合：

```
遊戲 tick → 有玩家的房間 → 隨機選物件或 NPC → 產生 ambient narrate
```

## 六、房間描述整合

### 6.1 描述文字規範

`rooms.json` 的 `description` 欄位中，可互動物件以 `〔物件名〕` 括起：

```json
{
  "id": "life_hall",
  "name": "浮生大廳",
  "description": "挑高的大廳中央懸著一盞〔六角宮燈〕，暖黃的光暈映在兩側的木柱上。櫃台後方掛著一面寫有【浮生】二字的〔匾額〕，墨跡蒼勁。角落立著一塊〔佈告欄〕，上頭釘滿了紙條。來往的旅人在此歇腳、打聽消息，或只是發呆望著門外。"
}
```

### 6.2 前端解析

`formatDesc()` 擴展，新增 `〔〕` 的解析規則。需將後端傳來的 `Objects` 陣列建成 name → id 的查找表，以便在描述解析時注入 `data-object-id`：

```javascript
// 快取當前房間物件（每次 view 更新時重建）
var currentRoomObjects = {}; // name → { id, actions }

function formatDesc(desc, objects) {
    if (!desc) return '';
    // 重建物件查找表
    currentRoomObjects = {};
    if (objects) {
        objects.forEach(function(o) { currentRoomObjects[o.name] = o; });
    }
    var safe = escapeHtml(desc);
    // 人物名：綠色
    safe = safe.replace(/【([^】]*)】/g,
        '<span class="desc-highlight">【$1】</span>');
    // 可互動物件：琥珀色，可點擊（僅當物件存在於 Objects 清單時才標記）
    safe = safe.replace(/〔([^〕]*)〕/g, function(match, name) {
        var obj = currentRoomObjects[name];
        if (obj) {
            return '<span class="desc-object" data-object-id="' + escapeHtml(obj.id)
                + '" data-object-name="' + escapeHtml(name)
                + '" role="button" tabindex="0">〔' + escapeHtml(name) + '〕</span>';
        }
        return '〔' + escapeHtml(name) + '〕'; // 無對應物件則純文字
    });
    return safe;
}
```

### 6.3 點擊即送觀看與 Log 其他動作列

描述區委派事件監聽，點擊 `.desc-object` 時調用 `sendObjectLook(objectEl)`，直接送出 `do_action`、`action: "Look"`。人物欄點擊 NPC 列同樣送出 Look。`action_result` 抵達後，main.js 在 append 觀看敘事後，依 `target_id` 從 `state.objects` 或 `state.entities` 取得其他動作，append 一行 `.log-object-actions`（【閱讀】【對話】【攻擊】等），並在 `#log` 上委派 `.log-object-action` 點擊以送出對應 `do_action`。

## 八、與現有系統的對接

### 8.1 插頭插座（決策 002）

非人物件等同決策 002 §2.2「建築／結構」和 §2.4「其他可互動物件」的房間制實現。每個物件的 `sockets` 就是它的插座清單，`do_action` 就是插頭。

### 8.2 NPC 活化系統

- **物件 ambient**：物件的環境敘事池可與 NPC 閒置動作混合，由同一個 idle tick 驅動。
- **NPC 看守**：NPC 的歸屬（員工身份）與物件的 `owner_guard` 對應，NPC 在場時負責「守護」物件。
- **NPC 反應**：偷竊被目擊時，在場 NPC 根據其 personality（`behaviors/*.json` 的 `theft_reaction`）產生反應。

### 8.3 事件日誌

所有物件互動（特別是偷竊、使用有主物件）記入 `event/journal`，供觀測坍縮和 NPC 記憶系統使用。

### 8.4 經濟系統

- `Use` 灶台 → 消耗食材（背包物品）→ 產出食物（新物品）
- `Taste` 酒甕 → 消耗鎂 → 品酒體驗
- `Take` 野生資源 → 物品進入背包
- `Open` 寶箱 → 鎂/物品掉落

## 九、有狀態物件（進階）

部分物件有內部狀態，影響可用動詞和互動結果：

### 9.1 範例：門

```json
{
  "id": "life_storage_door",
  "name": "庫房門",
  "owner": "inn",
  "sockets": ["Look", "Open", "Operate"],
  "state": "locked",
  "responses": {
    "Look": "一扇厚實的木門，門閂上掛著一把銅鎖。"
  },
  "state_responses": {
    "locked": {
      "Open": "門被鎖住了，紋絲不動。",
      "Operate": "你需要鑰匙才能開鎖。"
    },
    "closed": {
      "Open": "你推開了門。",
      "Operate": "你將門閂推上。"
    },
    "open": {
      "Look": "門大開著，門後是堆滿雜物的庫房。"
    }
  }
}
```

### 9.2 狀態轉移

```
locked --[有鑰匙+Operate]--> closed --[Open]--> open --[Operate]--> closed
                                                  ^                    |
                                                  |----[Open]----------+
```

狀態轉移邏輯由後端管理，前端只看到當下可用的動詞（不同狀態開放不同插座）。

## 十、實作路線

### 第一步：純觀看（最小可行）✅ 已完成

1. 建立 `data/room_objects.json`，5~8 房各 2~3 物件，`Look` / `Read` / `Smell` 等插座
2. 後端載入 `db/room_object.go`（含 `_說明` 等鍵的 JSON 解析）

4. `rooms.json` 描述加入 `〔〕` 標記，前端 `formatDesc` 解析為 `.desc-object`
5. **點擊即觀看**：點描述物件或人物欄 NPC 即送 Look，觀看敘事顯示於 Log
6. **Log 下一行其他動作**：main.js 在觀看後 append 【閱讀】【對話】【攻擊】等可點擊列（`.log-object-actions` / `.log-object-action`），人物欄取消手風琴

### 第二步：歸屬權 + 偷竊

1. `handleDoAction` 加入歸屬權判定流程（§4.2）
2. 「四下無人」判定函式
3. 偷竊事件寫入日誌
4. NPC 看守者的阻止反應（使用 NarrateMsg）
5. `data/room_objects.json` 補充 `owner_guard` / `use_blocked` / `use_steal`

### 第三步：有狀態物件

1. 物件狀態持久化（SQLite `room_object_states` 表）
2. 狀態轉移邏輯
3. 狀態感知的 Sockets 回傳（不同狀態開放不同動詞）

### 第四步：物件環境敘事

1. 物件 ambient 池接入 idle tick
2. 與 NPC 閒置動作混合，同一房間的物件和 NPC 交替產生環境敘事

### 第五步：互動產出

1. Use → 製作/產出（對接經濟系統）
2. Take → 採集（背包系統）
3. Open → 掉落（物品表）
