# 星盤迷霧與命盤機制規格

> **定位**：實作層。定義「迷霧」在前端 UI 與後端資料的具體行為規則，供 Cursor 實作時直接對齊。世界觀敘事見 [世界觀：Token降維與生命演化](世界觀：Token降維與生命演化.md)。
>
> **來源**：主管口述（2026-03-02 Gemini 對話 Turn 3、13、15），結合既有規格收斂。對齊 [狀態與星盤分頁規格](狀態與星盤分頁規格.md)、[人物屬性彙整](人物屬性彙整.md) §2.0、[361拓撲系統規格](361拓撲系統規格.md) §6。
>
> **原則**：不增加前端動畫或特效負擔；以**純文字**呈現迷霧狀態。

---

## 一、核心原則：名存實未知

> 主管原話（Turn 15）：「這只是解釋每個人的星盤內星位名稱都是顯示的，但是內容是未知的。」

### 1.1 定義

星盤上所有 361 個節點的**名稱恆顯**——二十主樞的名號（天極、震淵……）、五常邏輯閘的代號（起、承、轉、協、合）、十二微型狀態的代號（G01–G12），在 UI 上始終可見。

但節點的**功能性內容**——包括 Cost（阻力值）、通暢度標籤、屬性加成類型、貫通條件——在該節點**未被探明**之前，一律顯示為 `[未知]`。

### 1.2 與 activated_nodes 的關係

現有 `activated_nodes` 欄位（TEXT/JSON，預設 `["N000"]`）控制的是**貫通狀態**——節點是否已被打通。

「名存實未知」新增了一個語義層：**探明狀態**——節點的 Cost 與屬性是否已被玩家觀測到。

- **未探明**：名稱顯示，但 Cost 顯示 `[未知]`，適性顯示 `[未知]`，無法進行打通操作。
- **已探明**：名稱顯示，Cost 正常顯示（語意標籤 + 精確值），適性正常顯示，可進行打通操作。
- **已貫通**：節點 ID 存在於 `activated_nodes` 中。

> **實作預留**：第一版可沿用 `activated_nodes` 單一欄位。未來需區分「探明」與「貫通」時，可新增 `revealed_nodes`（TEXT/JSON）欄位，預設 `["N000"]`，記錄已透過 `[內視]` 探明的節點。N000 因預設貫通，亦預設探明。

## 二、迷霧的定義

### 2.1 迷霧是什麼

迷霧不是外物，而是**觀測盲區**。

世界觀層面：生之奇點（N000）激發後噴湧的氣體充滿了星盤空間，遮蔽了其餘 360 顆星的實相。見 [世界觀：Token降維與生命演化](世界觀：Token降維與生命演化.md) §五、§七。

機制層面：節點 ID 不在已探明清單（未來的 `revealed_nodes`）中，即處於「迷霧」狀態。前端渲染該節點時，顯示名稱但內容欄位以 `[未知]` 填充。

### 2.2 前端渲染規則

以下規則對齊 [狀態與星盤分頁規格](狀態與星盤分頁規格.md) §三：

#### 區塊二（二十主樞目錄）

| 欄位 | 未探明 | 已探明未貫通 | 已貫通 |
|------|--------|-------------|--------|
| 主樞名稱 | 顯示（如 `[天極] N001`）| 顯示 | 顯示 |
| 貫通進度 | `[未知]` | `x/17` | `x/17` |
| 適性 | `[未知]` | 語意標籤 + 精確值 | 語意標籤 + 精確值 |
| 操作按鈕 | `[迷霧籠罩]`（不可交互）| `[進入觀測]` | `[進入觀測]` |

#### 區塊三（觀測儀——藍點、綠點詳情）

| 欄位 | 未探明 | 已探明未貫通 | 已貫通 |
|------|--------|-------------|--------|
| 節點代號 | 顯示（如 `起 B01`）| 顯示 | 顯示 |
| Cost | `[未知]` | 語意標籤 + 精確值 | 語意標籤 + 精確值 |
| 狀態描述 | `氣雲遮蔽，內容觀測不可` | `未貫通` | `已貫通` |

### 2.3 探明操作：內視

探明節點的方式為 `[內視]`（已在 [人物屬性彙整](人物屬性彙整.md) §2.0 與 [361拓撲系統規格](361拓撲系統規格.md) §6 定義）：

- 消耗：氣脈或鎂（具體數值待定）。
- 效果：將目標節點加入 `revealed_nodes`，前端該節點的 Cost 與屬性從 `[未知]` 轉為實際值。
- 限制：只能探明與已探明／已貫通節點**直接相連**的節點（遵循拓撲連線規則）。

> **第一版**：不實作 `[內視]` 流程。全部主樞的探明狀態可先寫死（如預設全部已探明，或僅 N000 已探明）。此規格為未來實作的文檔預留。

## 三、命盤：SoulSeed 決定的專屬拓撲

> 主管原話（Turn 13）：「每個人的星位距離都不一樣，也有人戲稱這星盤是個人命盤，因為星系方位及距離，似乎是命中注定的。」

### 3.1 命盤 = SoulSeed 決定性生成的 760 條邊權

「命盤」是世界觀敘事對 SoulSeed → 760 Cost 這一機制的**文學稱謂**。

- **星位距離不同** ＝ 每條邊的 Cost 不同 ＝ SoulSeed 第 4–763 次 RNG 的結果。
- **方位不同** ＝ 哪些主樞 Cost 低（天生親近）、哪些 Cost 高（天生疏遠），因人而異。
- **命中注定** ＝ SoulSeed 創角即定，此後不變。

此機制已在 [人物屬性彙整](人物屬性彙整.md) §2.0 與 [361拓撲系統規格](361拓撲系統規格.md) §6.1 完整定義，無需新增後端邏輯。

### 3.2 萬人萬相的鐵律

> 主管原話（Turn 9）：「這違背了創角即生成 seed 的萬人萬相的設定。」

任何系統設計不得：

1. 為 SoulSeed 三軸賦予職業/流派/原型的語義映射。
2. 以預設模板覆蓋 SoulSeed 的決定性生成結果。
3. 讓後天操作改變 SoulSeed 或 760 Cost 的值（內盤不可變性）。

## 四、初次觀測敘事（創角 Onboarding）

> 主管原話（Turn 13）：「人們發現在初次激發生之奇點的時候能看到一堆星體，再次觀測卻發現除了生之奇點，其餘周遭都被生之奇點噴湧的氣體遮蔽。」

### 4.1 敘事流程（純文字系統日誌）

創角完成後，系統日誌可依序輸出以下文案（具體措辭待主管定稿）：

1. 止念狀態引入降維 Token 輻射。
2. 生之奇點（N000）被激發。
3. **瞬間瞥見**：意識穿透混沌，短暫看見 360 顆星體散布於無邊的內宇空間。
4. **氣體噴湧**：N000 噴發的氣體迅速蔓延，遮蔽了周遭一切。
5. **迷霧籠罩**：再次觀測時，除 N000 外，360 顆星的內容皆不可知。

### 4.2 不需要的東西

- **不需要**閃現動畫、粒子特效、Canvas/WebGL Shader。
- **不需要**計算或顯示「偏移率」「距離單位」等 Gemini 提議的數據。
- **不需要**讓玩家看到座標圖形——保持想像。

## 六、第一版實作建議

1. **最小改動**：在 [狀態與星盤分頁規格](狀態與星盤分頁規格.md) §3.2 的主樞目錄，增加 `[未知]` 作為適性的可能值。前端判斷：若節點不在已探明清單，則渲染 `[未知]`。
2. **資料面**：第一版可暫不新增 `revealed_nodes` 欄位。可在前端以 `activated_nodes` 同時代表「已探明 + 已貫通」，因為目前只有 N000 被激活。等 `[內視]` 機制實作時再拆分。
3. **文案面**：在星盤分頁頂部加入一行固定文案：`「周天 360 星，名存而實未知。唯有內視，方見其真。」`（措辭待主管定稿）
