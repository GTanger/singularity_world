## 一、後端

| 選項 | 簡述 | 優點 | 缺點 | 與約束對齊 |
| --- | --- | --- | --- | --- |
| **A. Node.js（Hono / Fastify）** | 單一語言 TypeScript/JavaScript，非阻塞 I/O，易與前端共型別 | 前後端同語系、生態豐富、WebSocket 成熟、迭代快 | 單執行緒（cluster 或多進程擴展）；萬名 NPC 需分片或定時 tick 設計 | 唯一真理 ✓；萬名 NPC 需架構設計（分區、tick 池）；無上架 ✓ |
| **B. Go（Echo / Gin）** | 編譯型、輕量協程、高並發 | 並發原生、部署單二進位、記憶體占用低、適合長連與 tick | 生態與遊戲後端範例較少；與前端不同語系 | 唯一真理 ✓；萬名 NPC 協程/worker 友善 ✓；無上架 ✓ |
| **C. Python（FastAPI / asyncio）** | 非同步框架、開發效率高、邏輯與數值易讀 | 詞盤／屬性等規則易寫易改、AI/腳本可擴充 | GIL 限制、高並發需多進程；WebSocket 與長連需額外設計 | 唯一真理 ✓；萬名 NPC 需多進程或外掛服務；無上架 ✓ |
| **D. Rust（Axum / Actix）** | 零成本抽象、記憶體安全、極高效能 | 效能與控制力最佳、適合寫遊戲伺服器核心 | 學習曲線陡、迭代慢、與前端不同語系 | 唯一真理 ✓；萬名 NPC 效能最佳 ✓；無上架 ✓ |
| **E. Cloudflare Workers + Durable Objects** | 邊緣無狀態 + DO 有狀態、按請求計費 | 全球邊緣、自動擴展、D1/SQLite 內建；DO 可一實體一 DO 或分區 | 執行時限制（CPU/記憶體）、需適配；萬名 DO 需設計分區與冷啟動 | 唯一真理 ✓；萬名 NPC 以 DO 分區模擬 ✓；無上架 ✓（自架或 CF） |

**組員建議**：**A 或 B**。A 利於前後端同棧、快速迭代與詞盤／屬性細規落地；B 利於長期並發與萬名 NPC tick、部署簡單。若團隊以 JS/TS 為主選 A；若重視並發與單機效能選 B。

## 二、前端

| 選項 | 簡述 | 優點 | 缺點 | 與約束對齊 |
| --- | --- | --- | --- | --- |
| **A. React + Vite** | SPA 或 SSG，元件化、生態大 | 組件化、狀態管理成熟、PWA 易做、觸控事件支援好 | bundle 體積需控制；重文字輕畫面下可精簡 | 手機指觸 ✓；無上架→PWA ✓；僅顯示窗口 ✓ |
| **B. Vue 3 + Vite** | 漸進式、模板語法清晰 | 上手快、文檔好、體積較小、PWA 支援 | 遊戲 UI 範例較少 | 同 A |
| **C. Svelte / SvelteKit** | 編譯時、少虛擬 DOM | 產物小、程式簡潔、適合輕量 UI | 生態較小、遊戲案例少 | 同 A，且更輕量 |
| **D. Vanilla JS + 輕量庫** | 無框架或僅用虛擬 DOM 小庫 | 最小依賴、完全掌控、MUD 式一頁即可 | 大型化後結構需自律；狀態與即時更新需自建 | 同 A；最省資源 |
| **E. React Native / Expo** | 原生殼、JS 開發 | 真原生手勢與效能、若日後要上架可沿用 | 無上架時收益有限；建置與除錯較重 | 手機 ✓；無上架下非必要 |

**組員建議**：**A 或 D**。A 利於協作與後續擴充（圓＋中文字、拖曳地圖、視野內實體）；D 利於極簡與「重文字輕畫面」、載入最快。首版可 A（Vite + React）做 PWA，前端僅為顯示窗口、資料全走後端。

## 三、資料庫

| 選項 | 簡述 | 優點 | 缺點 | 與約束對齊 |
| --- | --- | --- | --- | --- |
| **A. PostgreSQL** | 關聯式、ACID、功能完整 | 複雜查詢、JSON 支援、萬名 NPC 可存狀態與關聯、擴展方案多 | 需部署與維護；單機寫入有上限 | 唯一真理 ✓；NPC／世界／詞盤／經濟皆可建模 ✓ |
| **B. SQLite** | 單檔、嵌入式、零配置 | 部署簡單、適合單機或小團隊、詞盤／屬性表易建 | 並發寫入受限、大規模需分庫或讀寫分離 | 唯一真理 ✓；萬名 NPC 若以讀為主、寫分片可考慮 ✓ |
| **C. Redis + 持久化** | 記憶體為主、可 AOF/RDB | 快取、session、即時狀態、訂閱發布 | 複雜查詢弱、持久化非主場景；宜與關聯式搭配 | 唯一真理（若狀態以 Redis 為準）✓；常與 A/B 搭配 |
| **D. Cloudflare D1** | 邊緣 SQLite、與 Workers 整合 | 與 CF Workers 同棧、無伺服器、全球複製可選 | 綁定 CF、SQLite 限制同 B | 若後端選 E 則自然搭配 ✓ |
| **E. 記憶體 + 定時持久化** | 狀態在記憶體、定期寫入檔案或 DB | 極快、原型簡單、萬名 NPC tick 友好 | 重啟需恢復、需備援與日誌設計 | 唯一真理需嚴謹設計 ✓；適合作為遊戲狀態層＋DB 持久化 |

**組員建議**：**A 或 B**。A 利於長期、複雜查詢與萬名 NPC 關聯；B 利於快速啟動與自架單機。若後端選 Cloudflare 則 D。可採用 **E + A/B**：運行時狀態在記憶體（或 Redis），PostgreSQL/SQLite 做權威持久化與查詢。

## 四、連線方式

| 選項 | 簡述 | 優點 | 缺點 | 與約束對齊 |
| --- | --- | --- | --- | --- |
| **A. WebSocket 雙向** | 長連、雙向即時 | 世界更新、移動、戰鬥 log、視野內實體可即時推；手機瀏覽器支援佳 | 需處理斷線重連、背壓、心跳 | 唯一真理（後端推狀態）✓；萬名 NPC 僅推視野內 ✓；手機 ✓ |
| **B. Server-Sent Events (SSE)** | 伺服器單向推送 | 實現簡單、自動重連、HTTP 相容 | 客戶端僅能 HTTP 上送、需另通道送操作 | 唯一真理 ✓；若操作用 REST 則可 ✓ |
| **C. REST + 輪詢** | 無長連、週期請求 | 最簡單、防火牆友善、除錯容易 | 即時性差、耗電與流量較高 | 唯一真理 ✓；即時感較弱 |
| **D. WebSocket + REST 混用** | WS 負責世界／即時流，REST 負責動作與查詢 | 職責清晰、操作可冪等、易除錯與重試 | 需維護兩套端點與認證 | 唯一真理 ✓；推薦 ✓ |
| **E. 長輪詢** | 請求掛起直到有資料或逾時 | 無需 WS 基礎設施、穿透性好 | 延遲與連線數限制、實作較繁 | 唯一真理 ✓；次選 |

**組員建議**：**A 或 D**。世界永遠在線、視野內實體與移動需即時更新，WebSocket 最貼合；動作（移動目標、插頭／插座）可走 REST 或經由 WS 封包。**D（WS 推世界 + REST 動作）** 利於介面清晰與後端唯一真理。

## 七、未定版：後 C 前 A — 從「註解」開始學著看懂 AI 寫的

> **狀態**：未定版，依主管自白重寫，供討論與決斷。  
> **組合**：後端 **C（Python）** + 前端 **A（React + Vite）**。  
> **核心**：主管仍驅動 AI 寫代碼，但希望**試著學著看懂 AI 寫的內容**；**先從註解要求**做起。

---

### 7.1 主管自白（背景與動機）

- 主管**只學過微米程度的 Python**，程式非本業，但專案由主管決策與驅動。
- **過往 RUST 專案**：AI 一直鬼打牆、修不好 BUG，主管**看不懂代碼寫什麼**，只能一直下指令、無法自己判斷或微調，體驗很挫折。
- **現在**：仍由主管驅動 AI 寫代碼，但**想嘗試學著看懂 AI 寫的內容**，而不是只能「下指令→等結果→再下指令」。
- **第一步**：先從**註解要求**開始——**每行代碼後面都必須註解本行代碼作用是什麼**。這樣主管可以依註解逐行對照，慢慢建立「這行在幹嘛」的對應，進而能判斷 AI 是否寫對、該修哪裡、該怎麼下指令。

此建議書以「支援主管從註解開始、逐步能讀碼」為目標，並避免重蹈「語系過於艱深、主管完全無法跟上看懂」的覆轍。

---

### 7.2 為何建議「後 C（Python）」配合此目標

| 面向 | 說明 |
| --- | --- |
| **你已有微米級 Python** | 不是從零開始；註解 + 少量關鍵字（`if`/`for`/`def`/`return`）就能對上「這行在幹嘛」，比完全陌生的語系容易跟上。 |
| **RUST 的教訓** | Rust 語法與所有權、生命週期對非專職開發門檻高；主管當時看不懂，導致無法參與除錯與方向微調。**選一個主管有機會讀懂的語系**，再疊加註解，才能把「學著看懂」落到實處。 |
| **Python 的讀碼門檻** | 語句短、少括號、貼近自然語序；搭配**每行註解**，主管可先讀註解理解意圖，再對照該行代碼，逐步把「註解↔代碼」連起來，等於有一條**學習橋樑**。 |
| **AI 產出可約束** | 用專案規範與 prompt 明確要求「每行（或每邏輯行）後加 `#` 中文註解本行作用」；AI 產出可符合，主管審閱時才有材料可讀。 |
| **詞盤／屬性規則** | 本專案後端大量是表驅動與規則（詞盤細規、人物屬性細規）；Python 寫這類邏輯簡短，一行一意多，**註解對齊容易**，主管也較易對照規格與代碼是否一致。 |

因此：**選 Python 不是只因為「可讀」**，而是因為**主管已有微米級基礎 + 想從註解開始學著看懂 + 要避免 RUST 式完全看不懂**；Python + 每行註解能同時滿足「技術可行」與「主管可參與讀碼」兩件事。

---

### 7.3 「每行註解」— 角色與落實

- **角色**：註解是**主管的閱讀輔助**，也是**與 AI 協作時的契約**——「請你寫的每一行都說明在做什麼，我才能對照著看」。
- **落實**：
  - **編碼規範**：後端 Python 碼須在每行（或每個邏輯單位）後以 `#` 註解該行／該塊作用；註解以**中文**為佳，方便主管閱讀。
  - **AI 約定**：在專案說明或 prompt 中寫明上述要求；組員（或 AI）產出時一律帶註解，**不改碼不改註**（改碼必改註）。
  - **粒度**：若某行非常 trivial（如單獨 `return`），可與上一行合併說明，避免註解變成噪音；其餘以「主管能看懂這行在幹嘛」為準。
- **效果**：主管不必先成為 Python 專家；**先讀註解、再對照代碼**，就能判斷「這段是不是在做我要的」「BUG 可能出在哪一段」，下指令時更有依據，而不是只能一直重複描述問題。

---

### 7.4 後 C 前 A 技術可行性（簡要）

| 維度 | 說明 |
| --- | --- |
| **前後端** | React 以 HTTP/WebSocket 呼叫 Python 後端（如 FastAPI），介面 JSON，**可行**；前後端契約靠 API 規格（OpenAPI）與型別定義維持。 |
| **萬名 NPC** | Python GIL 限制單進程 CPU 並行；可採 **asyncio 分區 tick**、**多進程**、或外掛 worker。**可行但需架構設計**；首版可先千級 NPC 再擴，降低風險。 |
| **詞盤／屬性** | 規則與表驅動多，Python 易寫易對照細規，**與後 C 契合**。 |

技術細節與風險緩解同 §一～§三；此處不重複，僅強調：後 C 前 A 在技術上**可行**，與「主管從註解開始學著看懂」的目標**不衝突**，且能互相支援。

### 7.5 建議書結論（未定版）

- **後 C 前 A** 在此專案下**可行**，且能配合主管「從註解開始、學著看懂 AI 寫的」的目標；Python 門檻相對低、你已有微米級基礎，加上每行註解，可避免重蹈 RUST 專案「完全看不懂、只能一直下指令」的狀況。
- **每行註解**是**先決條件**：不論最終是否拍板後 C，只要主管希望「試著看懂 AI 寫的」，都建議在**後端**採用「每行（或每邏輯單位）註解本行作用」的規範；若選 Python，此規範落實成本低、可讀收益高。
- **若主管決斷**：採 **後 C 前 A + 每行註解**，則建議將「後端 Python 碼須帶行尾／塊註解（中文、說明該行／該塊作用）」寫入專案規範或 CONTRIBUTING，並在與 AI 協作時一貫要求；前端 React 若主管暫不參與讀碼，可先不強制同級註解，待後端讀順再擴及前端亦可。

---

### 7.6 若不鎖定後 C 前 A，依自白的推薦

在**不預設**後端一定是 Python、前端一定是 React 的前提下，依你剛剛的自白（微米級 Python、RUST 看不懂只能下指令、想學著看懂 AI 寫的、先從註解要求），組員的推薦如下。

- **後端**：仍推薦 **C（Python）**。  
  理由：你**唯一有微米級基礎**的語系就是 Python；「學著看懂」需要有一點起點，否則純靠註解從零對照，負擔會很大。RUST 的教訓是語系門檻太高、完全跟不上；Go / Node 你沒有同樣的微米級基礎，註解可以幫上忙，但「註解 + 微米級 Python」是**對你最友善**的組合。若你願意接受「從零開始對照註解學另一門語系」，則 **A（Node/TS）** 可當次選——前後端同語系、生態大，但你就沒有「那一點點先修」的優勢。

- **前端**：仍推薦 **A（React + Vite）**。  
  理由：自白裡你關注的是「看懂 AI 寫的」、當時卡住的是 RUST 後端，未必要同時讀前端。前端選 A 主要是**技術契合**（手機、PWA、顯示窗口、生態成熟）；若日後你也想參與看前端碼，再對前端加註解要求即可，不必一開始就鎖定「前端也要主管能懂」。

- **先決條件不變**：不論後端最後選 C 或 A 或 B，只要你想「試著看懂 AI 寫的」，**每行（或每邏輯單位）註解本行作用**都建議當成**專案規範**；否則換什麼語系都難以擺脫「只能一直下指令」的循環。

**一句話**：不鎖定後 C 前 A 的話，依自白我仍推薦 **後 C 前 A**——因為那是對「主管從註解開始學著看懂」最友善的組合；若你有其他考量（例如更在意並發／單二進位部署而想選 Go），可改 **後 B 前 A**，但請務必保留**每行註解**，並接受讀碼時沒有微米級 Python 的起點。

## 八、主管拍板：技術棧與架構

主管已決斷技術棧與架構，完整內容見 **[決策 004：技術棧與架構](../decisions/004_tech_stack_architecture.md)**。摘要如下。

| 層 | 選型 | 要點 |
| --- | --- | --- |
| **伺服器** | **Go** | 遊戲主迴圈、視野內 NPC 即時行為；量子坍縮系統（惰性求值 + 事件日誌）；經濟／政治／生態引擎（獨立 goroutine、各自 tick rate）；WebSocket 最多 10 人；SQLite 持久化 |
| **前端** | **網頁（HTML + CSS + JavaScript）** | WebSocket 連伺服器；Canvas 2D 俯視地圖 + 符號移動；HTML 原生富文字處理詞盤、對話、敘事；不需安裝，開瀏覽器即玩 |
| **資料庫** | **SQLite** | NPC 狀態、事件日誌、經濟數據、玩家存檔；十人規模綽綽有餘 |

**架構核心**：九萬 NPC 平時不跑，只存狀態 + 上次觀測時間；觀測發生時從事件日誌回推當前狀態（延遲坍縮）；視野內 NPC 才進入即時模擬；全域系統（經濟、政治、生態）背景跑，產出事件流。

設計與實作依此技術棧與架構進行。
